## 들어가며

면접에서 _"AI 도구 쓰시나요?"_ 라는 질문을 받은 적이 있다. 2026년에 안 쓰는 프론트엔드를 찾는 게 더 어렵다. 진짜 차이는 도구를 쓰느냐가 아니라, 도구가 잘 일할 수 있는 환경을 미리 깔아두었느냐다.

아내와 둘이서 신혼부부용 갤러리·가계부 서비스 UANDI를 만들면서, 우리는 코드를 짜기 전에 AI가 일할 환경부터 만들었다. 이 글은 그 워크플로우에 대한 기록이다.

## 1. 문제: AI에게 설명하는 시간이 코드 짜는 시간보다 길어졌다

Claude Code를 본격 도입하고 한 달이 지나자 이상한 일이 일어났다.

- "이 컴포넌트는 어떤 디자인 규칙을 따라?" — 매번 설명
- "Photo 타입은 어떤 필드를 가져?" — 매번 설명
- AI가 짜준 코드의 70%는 맞고 30%는 우리 컨벤션과 미묘하게 어긋남 → 검수에 더 많은 시간

문제는 모델 성능이 아니라 **컨텍스트의 휘발성**이었다. 매번 채팅창에 같은 설명을 풀어내고 있었다.

## 2. 해결: 모든 결정을 마크다운에 박아둔다

UANDI의 `/docs` 폴더는 8개 문서로 구성된다. (출처: `00-overview.md`)

```
docs/
├── 00-overview.md         # 비전, P0/P1 기능 범위
├── 01-tech-stack.md       # Next.js 15, React 19, Firebase
├── 02-design-system.md    # 컬러, 컴포넌트 규칙
├── 03-domain-models.md    # Firestore 스키마 = TS 타입
├── 04-auth-flow.md        # 인증 상태 머신
├── 05-testing-strategy.md # E2E Test-First 사이클
├── 06-docs-site.md        # 명세 사이트 자체 명세
├── 07-extension.md        # 크롬 익스텐션 명세
└── pages/{페이지별}.md
```

읽는 순서까지 명시되어 있다. **사람에게 좋은 온보딩 문서가 AI에게도 좋은 컨텍스트**다.

### 명세를 코드보다 먼저 쓴다는 의미

가계부 카테고리를 예로 들어보자. 보통은 컴포넌트 안에 임시로 박은 enum이 여러 곳으로 흩어지다가 미묘하게 어긋난다. UANDI에서는 거꾸로 가게 했다.

```ts
// docs/03-domain-models.md 에 먼저 정의
export const EXPENSE_CATEGORIES = [
  '식비',
  '교통',
  '쇼핑',
  '의료',
  '문화/여가',
  // ... 총 10개
] as const;
```

명세는 모든 코드와 AI 프롬프트가 따라야 할 단 하나의 기준점이다.

## 3. AI를 위한 별도 사이트를 만들었다 — `apps/docs`

마크다운을 폴더에 두는 것까지가 1단계라면, 2단계는 **명세 전용 Next.js 앱을 띄운 것**이다. (출처: `06-docs-site.md`)

### 왜 GitHub에서 그냥 보지 않고 별도 앱으로 띄웠나

**(1) AI 도구의 접근성** — `https://docs.uandi.app/domain-models` 같은 안정적인 URL이 있으면 AI 어시스턴트가 URL을 직접 읽어 컨텍스트로 사용할 수 있다. GitHub raw URL은 캐시·렌더링 모두 AI 친화적이지 않다.

**(2) 비개발자 협업자(아내)의 접근성** — `.md` 파일을 GitHub에서 보는 건 비개발자에게 진입 장벽이 높다. 사람이 읽기 좋은 사이트가 되어야 _"오늘 여기까지 만들었어"_ 가 가능해진다.

**(3) 자기 검열** — 자기 글을 사이트에서 다시 읽어보면 마크다운 에디터에서 못 잡던 누락이 보인다.

### 스택 (본 서비스와 동일하게)

| 영역          | 선택                                       |
| ------------- | ------------------------------------------ |
| 프레임워크    | Next.js 15 (`apps/web`과 동일)             |
| 스타일        | Tailwind 4 + `@tailwindcss/typography`     |
| 마크다운      | unified + remark + rehype                  |
| 코드 하이라이팅 | rehype-shiki (shadcn 공식 사이트와 동일) |
| 라우팅        | `00-overview.md` → `/overview` (번호 prefix는 정렬용) |

## 4. AI에게 컨텍스트를 주는 방식이 바뀌었다

**Before**

> "사진 갤러리 페이지 만들어줘. React Query로 페이지네이션, Tailwind로 그리드, 모바일 우선."

→ 동작은 하지만 우리 코드베이스와 어긋나는 코드.

**After**

> "다음 문서를 읽고 `/photos`를 구현해줘.
>
> - `docs/03-domain-models.md` (Photo 타입)
> - `docs/02-design-system.md` (컴포넌트 규칙)
> - `docs/pages/03-photo-gallery.md` (사용자 스토리)
> - `docs/05-testing-strategy.md` (Test-First)
>
> 사용자 스토리를 E2E 테스트로 먼저 변환한 뒤, 통과하는 최소 구현을 만들어줘."

핵심은 **반복 가능성**이다. 명세 없이 AI를 쓰면 같은 프롬프트도 매번 다른 결과가 나온다. 그건 도구가 아니라 복권이다. 명세 기반 프롬프트는 같은 품질의 결과를 반복해서 낸다.

## 5. 명세와 코드를 어긋나지 않게 만드는 규칙

명세 워크플로우의 가장 큰 함정은 시간이 지나면서 명세와 코드가 어긋나는 것이다. **명세가 거짓말쟁이가 되면 AI도 거짓말 위에서 일하게 된다.**

### 규칙 1. 도메인 타입은 명세와 글자 단위로 일치

`docs/03-domain-models.md`의 타입 정의와 실제 TS 타입을 일치시킨다.

### 규칙 2. 사용자 스토리 = E2E 테스트 이름

`05-testing-strategy.md`의 핵심 규칙은 **사용자 스토리를 한국어 문장 그대로 테스트 이름으로 쓰는 것**이다.

```
✓ '지출 내역을 추가하면 목록에 표시된다'
✓ '금액을 입력하지 않으면 저장 버튼이 비활성화된다'
✗ 'test cashbook form'  ← 나쁜 예
```

명세 → 테스트 → 코드를 자연어 단위로 추적할 수 있다는 뜻이다. AI도 사람도 같은 단위로 생각하게 된다.

## 6. 정리

AI를 잘 쓰는 사람과 못 쓰는 사람의 차이는 **모델 선택이 아니라 컨텍스트 설계**다.

_"AI 도구 쓰세요?"_ 는 의미 없는 질문이 됐다. 의미 있는 질문은 이제 이렇다.

- AI가 같은 품질로 일할 수 있도록 무엇을 미리 깔아두었나요?
- 명세와 코드가 어긋나지 않게 하는 규칙은 무엇인가요?
- 비개발자 협업자가 같은 컨텍스트를 공유할 수 있나요?

이 세 질문에 답할 수 있는 사람이 **"AI를 쓰는 개발자"가 아니라 "AI를 도구화한 개발자"** 다. 다음 편에서는 이 명세 인프라 위에서 만든 첫 AI 기능, **크롬 익스텐션의 자연어 가계부 입력**으로 이어진다.
