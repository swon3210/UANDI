## 역할

구현 사이클의 ④~⑤ 단계에서 사용한다.
작성된 UI 컴포넌트와 Storybook 스토리를 분석하고, 유저에게 **구조화된 리뷰 보고서**를 출력한다.
코드를 수정하지 않는다 — 유저 피드백을 받은 후에만 수정한다.

## 입력

`$ARGUMENTS` — 리뷰할 컴포넌트 디렉토리 경로 (예: `apps/web/src/components/photos`)

인자가 없으면 최근 작업한 컴포넌트 디렉토리를 자동으로 탐색한다.

## 실행 순서

### 1단계: 스토리 파일 수집

- 지정된 디렉토리에서 `*.stories.tsx` 파일을 모두 찾는다
- 각 스토리 파일과 대응하는 컴포넌트 파일을 읽는다

### 2단계: 컴포넌트 분석

각 컴포넌트에 대해 다음을 분석한다:

| 항목 | 내용 |
|---|---|
| **Props** | 타입, 필수/선택, 기본값 |
| **스토리 커버리지** | 기본 상태, 엣지 케이스(빈 데이터, 로딩, 긴 텍스트, 에러 등) |
| **의존성** | `@uandi/ui` 컴포넌트, 외부 라이브러리, 아이콘 |
| **반응형** | 모바일/태블릿/데스크탑 대응 여부 |

### 3단계: 갭 검사

- 스토리가 없는 컴포넌트가 있는지
- 중요한 엣지 케이스 스토리가 빠졌는지
- 접근성(a11y) 고려 사항이 있는지

### 4단계: Storybook 실행 안내

유저가 직접 확인할 수 있도록 실행 명령어를 안내한다.

## 출력 형식

```markdown
# /storybook-review: [디렉토리명]

## 컴포넌트 목록

| 컴포넌트 | 스토리 수 | 커버리지 |
|---|---|---|
| ComponentA | 3 | ✅ 기본, 빈 상태, 로딩 |
| ComponentB | 2 | ⚠️ 에러 상태 누락 |

## 컴포넌트 상세

### ComponentA
- **파일**: `path/to/ComponentA.tsx`
- **Props**: `prop1: string`, `prop2?: boolean`
- **스토리**: Default, Empty, Loading
- **의존성**: Button, Skeleton (from @uandi/ui)

### ComponentB
...

## 누락 항목 (있는 경우)
- [ ] ComponentC에 스토리 파일 없음
- [ ] ComponentA 에러 상태 스토리 추가 권장

승인하시면 페이지 조립 단계로 진행합니다.
```

## 중요

- 코드를 수정하지 않는다 (보고서만 출력)
- 유저가 "승인" 또는 "진행"을 명시하면 다음 단계(페이지 조립)로 넘어간다
- 유저가 수정을 요청하면 해당 컴포넌트/스토리를 수정한 뒤 다시 보고한다
